# Identifier Policy

The Math Translation Layer treats provenance as part of the product. Every durable run, source document, and translation event should have a stable identifier.

## Current IDs

### Run UUID

Generated by the TTS workflow at runtime:

```text
runUuid
```

Scope: one workflow execution.

Stored in:

```text
workflow_output/logs/<run-id>/summary.json
workflow_output/logs/<run-id>/<paper>.log
workflow_output/logs/<run-id>/<paper>.translation-events.json
```

### Document UUID

Generated from:

```text
absolute source path + source SHA-256
```

Scope: one exact file content at one path.

Stored as:

```text
documentUuid
sourceSha256
```

### Translation Event UUID

Generated from:

```text
documentUuid + equation span + equation source + renderer + mode
```

Scope: one translated math block inside one exact document.

Stored as:

```text
uuid
event
runUuid
documentUuid
```

### Browser Structural Event ID

Generated in the browser overlay from:

```text
equation source + structural token map
```

Stored in DOM attributes:

```html
data-mtl-event-id="mtl-struct-..."
data-mtl-event="structural-equation-map"
data-mtl-review-priority="high"
data-mtl-review="{...json...}"
```

The browser event ID is intentionally short and deterministic for client-side use. The TTS workflow UUIDs are stronger provenance IDs for archival records.

## Rule

Never overwrite or reinterpret an existing event ID. If the translation changes, generate a new event and link it back to the prior event as a repair/revision in a later registry.

## Next Registry Targets

The same policy should be extended to:

- paper UUID
- claim UUID
- theorem/proof UUID
- prediction UUID
- NLP review UUID
- human repair UUID
- release/build UUID
